Design: BILTIQ-002 — Multi-target fan-out

Approach

Extend the Increment 1 engine — no new tool, no data-model change. Four coordinated edits plus one new module:

  1. Planner emits one run_research_pipeline task per target. It accepts an explicit list verbatim and expands a descriptive goal from its own knowledge (no web), capping at config.gates-sibling ORCH_MAX_TARGETS. Excess targets are dropped and noted.
  2. Engine execute step changes from a sequential for loop to a bounded-concurrency pool (cap ORCH_FANOUT_CONCURRENCY): at most K tasks in flight; each still runs executeTask (its own gate + events). Replan/grade/termination unchanged.
  3. Aggregation (new aggregate.ts): a pure rankResults that sorts TaskResult[] best-first by pitch score, pushing !ok/low-confidence results to the bottom. The engine ranks before calling the synthesizer, so the final answer is targets-best-first.
  4. Goal grader prompt becomes coverage-aware: judge "M of N targets have confident pitches", not "one perfect output" — fixing the over-strict behavior the Inc 1 e2e surfaced.

Alternatives Considered

  • Dedicated fan_out tool — rejected: duplicates run/gate/persist the engine owns; child runs bypass the gate/event model.
  • Planner-inferred lists only — rejected: no explicit fast path; all lists carry hallucination risk.
  • Unbounded parallelism — rejected: each pipeline ≈12 LLM calls; would overload the endpoint and blow cost.

Risks & Rollout

  • Planner-expanded list stale/wrong → flag expanded lists; explicit lists bypass.
  • Concurrency overwhelms endpoint → bounded pool + max-targets cap, both env-tunable.
  • Partial fan-out → aggregate surfaces successes; partial=true, honest coverage report.
  • Rollout: additive/behavioral; single-target goals still produce one task and behave as Inc 1. Setting ORCH_FANOUT_CONCURRENCY=1 restores sequential execution.
