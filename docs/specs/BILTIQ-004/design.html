Design: BILTIQ-004 — Company-level chat

Approach

Extend the existing per-run chat pattern (routes/runs.ts /:id/chat: build context → callText → persist history) up to the company level. A new CompanyChat collection stores history keyed by (projectId, normalized companyKey). A pure buildCompanyContext(runs) helper merges all of a company's runs (top signals, summaries, latest pitch) into one system context. New chat endpoints live on the projects router. Frontend adds a per-company "Chat" drawer on the project detail page, mirroring the existing run ChatDrawer.

Alternatives Considered

  • Store history on the Project doc as a map — rejected: unbounded growth on one doc; a keyed collection is cleaner.
  • Re-use /:id/chat with a synthetic merged run — rejected: muddies the run model; a dedicated aggregator is clearer.

Data / Compliance / Rollout

Additive: one new collection. Compliance unchanged — chat goes through callText (switchable adapter), no new external dep. Per-run chat untouched. Empty company → 400 / disabled button.

Risks

  • Large companies → big context: cap merged signals (e.g. top 12) in buildCompanyContext.
  • Company key drift vs grouping: reuse normalizeCompany for the key so chat + grouping agree.
