Approach

Name

Hybrid plan-then-adapt loop in a new backend/src/orchestrator2/ layer, with the fixed 6-phase pipeline wrapped as a single tool and orchestration state held in a new OrchestrationRun model.

Description

A new planner-orchestrator layer sits above the existing fixed pipeline. The user states a free-form goal; an LLM planner produces an ordered list of run_research_pipeline tasks; the engine executes each task as a gated child Run (reusing the existing ProspectOrchestrator verbatim), gates the result with the Phase A runGate primitive, replans the remaining tasks from what it learned, and goal-grades — looping until the grader returns met:true or maxIterations (default 6) is hit. The key structural decision is that the engine takes its planner, replanner, grader, and tool registry as injected dependencies, so the loop's control flow is unit-testable with mocks and no live LLM. The pipeline is wrapped, not forked: run_research_pipeline constructs the same ProspectOrchestrator the existing worker uses, so all six phases, their gates, and their events are inherited for free. Trade-off: a new BullMQ queue, a new Mongo model, and a new SSE channel are added (more surface area) in exchange for keeping the existing structured /new flow and its Run contract completely untouched — the dynamic layer is purely additive.

Alternatives Considered

#AlternativeWhy Rejected
1Extend the existing prospect-research BullMQ queue with an orchestration job typeCouples orchestration retries/concurrency to per-pipeline jobs and overloads the existing worker handler; a separate orchestration queue keeps lifecycles, concurrency, and failure handling independent. (Chosen: new queue.)
2ReAct (single model loop interleaving thought + tool calls)Harder to bound cost and to unit-test the control flow deterministically; an explicit plan→execute→replan loop gives a visible plan board and injectable steps. (Chosen: hybrid plan-then-adapt.)
3Pure plan-then-execute (no replan between tasks)Cannot adapt to what a task discovered (e.g. a failed gate), so it can't satisfy adaptive goals — the whole motivation for the increment. (Chosen: hybrid adds a replan step.)
4Fork / modify orchestrator/index.ts to accept a goal directlyBreaks the working fixed pipeline and the /new contract, and duplicates phase logic; wrapping it as a tool inherits all phases and gates unchanged. (Chosen: wrap as tool.)
5Embed orchestration fields (plan, iterations, goalMet) on the existing Run modelConflates two different lifecycles (one goal session vs. one pipeline execution) and would require a schema migration on a live collection; a distinct OrchestrationRun model linking children via childRunId avoids touching Run. (Chosen: new model.)

Data Model Changes

One new Mongoose model — OrchestrationRun — added in backend/src/db/models/orchestrationRun.ts. The existing Run model and collection are unchanged (no migration on live data). New shared types go in backend/src/types.ts.

// backend/src/types.ts (new)
type TaskStatus = 'pending' | 'running' | 'done' | 'failed' | 'skipped';

interface PlanTask {
  id: string;
  tool: 'run_research_pipeline';   // Inc 1: only this tool
  args: Record<string, unknown>;   // e.g. { lead: { company, url, linkedinUrl } }
  rationale: string;
  status: TaskStatus;
  childRunId?: string;             // links to a Run via run.runId
  gatePassed?: boolean;            // result of the Phase A result gate
  resultSummary?: string;          // short, LLM-readable
}

type OrchestrationStatus =
  | 'queued' | 'planning' | 'executing' | 'replanning'
  | 'grading' | 'completed' | 'failed';

interface OrchestrationEvent {
  type: 'orchestration_start' | 'plan_created' | 'plan_revised'
      | 'task_start' | 'task_complete' | 'task_failed'
      | 'gate_check' | 'goal_graded'
      | 'orchestration_complete' | 'orchestration_failed';
  taskId?: string; data?: unknown; message?: string; timestamp: Date;
}

interface OrchestrationRun {
  orchestrationId: string;
  goal: string;
  hints?: Record<string, unknown>;
  status: OrchestrationStatus;
  plan: PlanTask[];
  iterations: number;
  maxIterations: number;           // default 6
  goalMet: boolean;
  partial: boolean;                // true when capped before goal met (mirrors lowConfidence)
  finalAnswer?: string;
  events: OrchestrationEvent[];
  createdAt: Date; updatedAt: Date;
}

Child-Run linkage: when run_research_pipeline runs, it creates a Run via RunModel.create({ runId, lead, … }) (the same path as routes/leads.ts), executes ProspectOrchestrator, and persists the final state. The task records that runId on PlanTask.childRunId — so the plan board links to /runs/[id] and the existing run-detail view renders it. Per overview.md, a new data model is ADR-worthy: the dev writes ADR-0002 before build (additive model, no migration).

// Engine — injectable deps (the LLM-free testability contract):
interface EngineDeps {
  planner:   (goal: string, hints?: object) => Promise<PlanTask[]>;
  replanner: (goal: string, plan: PlanTask[], results: TaskResult[]) => Promise<PlanTask[]>;
  grader:    (goal: string, results: TaskResult[]) => Promise<{ met: boolean; reasoning: string; score: number }>;
  synthesizer: (goal: string, results: TaskResult[]) => Promise<string>;
  tools:     Record<string, Tool>;        // the TOOLS registry
  emit:      (e: OrchestrationEvent) => Promise<void>;
  maxIterations: number;
}

In tests the engine is constructed with mock planner/replanner/grader/tools that return canned values, so the loop's branching (stop-on-met, stop-on-cap, replan-applied, status transitions) is verified with zero LLM calls.

Result Gate Reuse & Injectability

How runGate (Phase A) is reused: after a task's child Run completes, the engine calls runGate(childRun, cfg) where the grader is gradePipelineResult — deterministic: pass = !!run.pitch && !run.lowConfidence. With maxRevisions bounded (Inc 1 may use 0 — grade once, no re-run, or 1 to allow one bounded re-run of the same pipeline task), this reuses the exact same control-flow helper the per-phase gates use. The gate result sets PlanTask.gatePassed, emits a gate_check event, and is fed into the replan so a failed gate influences the revised plan (spec AC6). No new gate machinery is written — GateResult, GateAttempt, and runGate all come from orchestrator/gate.ts.

Injectable-dependency shape (AC4): the engine never imports planner.ts/replanner.ts/goal-grader.ts/tools.ts directly — it receives them via EngineDeps (above). The worker wires the real LLM-backed implementations; the unit tests wire mocks. This is what makes "stop-on-met / stop-on-cap / replan-applied / status-transitions" testable without an LLM, and is the same dependency-injection pattern the Phase A runGate tests already use (mock grader/reviser).

Security & Compliance

  • Compliance mode on_prem_preferred: every LLM call (planner, replanner, goal-grader, synthesizer) routes through the existing switchable adapter llm/client.ts (callText/callWithTool). When configured for on-prem vLLM, no data crosses an external boundary — the dynamic layer adds no new external dependency, so no new ADR for cloud AI is required beyond the existing OpenAI-default ADR already covering the pipeline.
  • Bounded iterations (maxIterations default 6) cap LLM spend and prevent runaway loops — a cost and safety control.
  • Secret handling: all keys come from config.ts (env), never hardcoded. The worker is a separate process and reads config from env, not server module state — orchestration config (ORCH_MAX_ITERATIONS) follows the same rule as the Phase A gate config.
  • Input escaping: the user-supplied goal and hints are embedded in LLM prompts (planner/grader) and stored verbatim. They are not interpolated into Mongo queries (Mongoose parameterizes) and are rendered as text (not HTML) on the plan board. No SQL — Mongo only.
  • New Mongo model + new external-facing routes ⇒ ADR-0002 (data model) and a one-line threat-model note on the new endpoints, per AGENT_RULES.md.

Performance Considerations

Increment 1 runs tasks sequentially (parallelism is out of scope; the TOOLS registry + per-task structure leaves room for Inc 2 fan-out). Cost per orchestration = (sum of child pipeline runs) + at most maxIterations × (1 planner + 1 replanner + 1 grader) extra LLM calls, plus one synthesizer call. Each child pipeline run is already cost-bounded by the Phase A per-phase gates. The hot path is the loop body's three LLM calls per iteration; with the default cap of 6 that is ≤ 18 orchestration-level calls on top of the pipeline work. The result gate is deterministic (no tokens) on the happy path. SSE reuses the in-process runEventBus — no extra transport cost. Wall-clock is dominated by sequential pipeline runs (scraping latency), as today.

Rollback

Because the feature is additive and isolated to orchestrator2/, the new queue, the new model, and the new routes/pages: revert the PR (or feature-branch). No data migration to undo — OrchestrationRun is a separate collection; dropping it does not affect Run. The orchestration worker process can simply be left unstarted to disable the feature without a code change.
