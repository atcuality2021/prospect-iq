Problem

ProspectIQ runs a fixed 6-phase pipeline hardcoded in backend/src/orchestrator/index.ts. The user supplies a structured lead and always gets the same sequence — there is no planning step, so the system cannot satisfy goals that need a different or adaptive set of actions. We want a dynamic orchestrator: the user states a free-form goal, an LLM planner decides what to do, runs it through sub-agents, quality-checks each result, adapts the remaining plan from what it learns, and loops until the goal is met. The existing pipeline becomes one tool the planner can call.

Increment 1 builds the full agentic loop (plan → execute → gate → replan → grade → synthesize) with a single tool — run_research_pipeline — to de-risk the control flow, data model, and UI before widening the tool palette in later increments.

User Impact

A new /orchestrate page lets a user type a sales goal (e.g. "Research Stripe and draft an outreach email") instead of filling a structured lead form. They watch a live "plan board" as the orchestrator plans, runs gated pipeline tasks, replans, and produces a synthesized final answer. The existing structured /new flow is unchanged and remains available.

Acceptance Criteria

  • OPENAC1 — POST /api/orchestrations {goal, hints?} creates an OrchestrationRun (status queued), enqueues a BullMQ job on a new orchestration queue, and the /orchestrate page receives live OrchestrationEvents over SSE.
  • OPENAC2 — The engine produces an initial plan, executes each task as a gated child Run, runs a replan step between tasks, and terminates when the goal grader returns met:true OR iterations reaches maxIterations (default 6).
  • OPENAC3 — A run capped before completion ends with goalMet:false, partial:true, a non-empty best-effort finalAnswer, and a "goal partially met" badge on the UI.
  • OPENAC4 — The engine accepts its planner/replanner/grader/tools as injected dependencies and passes LLM-free unit tests for: stop-on-met, stop-on-cap (sets partial), replan output applied to the pending-task list, and correct task status transitions.
  • OPENAC5 — A child Run produced by a task is persisted and linked via PlanTask.childRunId, and is reachable from the plan board (link to /runs/[id]).
  • OPENAC6 — Each task result is gated via the Phase A runGate primitive (pass = child Run has a pitch and is not lowConfidence); a failed gate is visible in the events and influences the replan.
  • OPENAC7 — Adding a tool in a later increment requires only a new TOOLS registry entry plus a planner-schema enum value — no engine.ts changes (verified by code structure / review).

Out of Scope

  • fan_out parallel sub-agents over entity lists (Increment 2).
  • Granular per-phase tools — scrape/verify/synthesize/match/pitch/reflect as individual tools (Increment 3).
  • web_search discovery tool (Increment 4).
  • Parallel task execution (Increment 1 runs tasks sequentially; engine is structured to allow parallelism later).
  • Human-in-the-loop plan editing / approval before execution; auth; multi-user.

Open Questions

  • Resolved: planner style = hybrid plan + adapt; tool = run_research_pipeline only; provider per on_prem_preferred.
  • Pending design call (for biltiq-code-architect): does the orchestration use a new BullMQ queue or extend the existing prospect-research queue with a job type? (Spec assumes a new orchestration queue.)
  • Pending: SSE vs. polling for the plan board (spec leans SSE, reusing events/emitter.ts keyed orchestration:<id>).
