Reflect: BILTIQ-001 — Dynamic Orchestrator, Increment 1

Outcome: shipped. 11 atomic steps, 62 LLM-free unit tests, code-reviewer approved, live e2e passed, on PR #1.

What went well

  • Injectable EngineDeps was the key decision. Taking planner/replanner/grader/synthesizer/tools as injected dependencies made the entire control loop (stop-on-met, stop-on-cap→partial, replan-applied, status transitions) unit-testable with zero LLM/DB. That's why 62 tests run in milliseconds.
  • The BiltIQ gates earned their cost. The biltiq-code-architect codebase scan caught the hard Phase-A→BILTIQ-001 dependency the markdown specs never encoded. The plan-reviewer reverse-coverage gate caught two ADRs + a regression file that had drifted into the plan without design review.
  • Wrapping over forking. The pipeline became a tool (run_research_pipeline) constructing the existing ProspectOrchestrator; reused runEventBus and the worker-persist shape verbatim. No duplication.
  • The e2e proved the honest-failure path. On real data the result gate failed two low-confidence child runs, the loop capped at 3 and returned partial=true with a synthesis that openly said no viable email exists — exactly the design intent, not a bug.

What broke / friction

  • Goal grader too strict. The e2e showed it rejected a decent score-45 pitch and demanded a single perfect output, which made the replanner re-run research and degrade to 0 signals. Root-caused as a prompt issue; fixed in BILTIQ-002 (coverage-aware grading).
  • as const schema widening. Extracting the planner's tool schema to a const widened type:'object' and froze arrays, breaking the callWithTool param type. Typecheck caught it; fixed by annotating UnifiedTool['schema']. Lesson: inline literals get contextual typing that extracted consts lose.
  • Specs didn't encode cross-task dependencies. The markdown loop-engineering spec and the BILTIQ-001 spec both existed without recording that one blocks the other — only the architect's disk scan surfaced it.

Anti-patterns observed (and handled)

  • AP#6 (dead code) — caught in build. gradePipelineResult was initially only called from tests. Wired it into tools.ts (tool ok = gate.pass) so it has a real production call path.
  • AP#1 (duplication) — avoided by design. Reused engine/gate/event/persist; the reviewer confirmed no fork.
  • AP#7 (residue) — two instances cleaned. A pre-existing stray next.config.ts.bak was found and git rm'd; the temporary e2e harness was deleted after use, never committed.
  • Review finding fixed, not deferred. Engine skipped final-iteration replan so capped runs don't persist dangling pending tasks.

Carry-forward

  • Grader strictness is a tuning surface → natural first target for Loop 4 (hill-climbing) once traces exist (Phase B).
  • HTTP+queue+worker path is typecheck-verified but only the engine path was e2e'd — do a manual /orchestrate click-through before merging PR #1.
  • Record cross-task dependencies in spec front-matter so the architect isn't the only safety net.
