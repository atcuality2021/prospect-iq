⚠ Prerequisite (BLOCKER from design scan): Phase A — L2 Verification Gates must be merged first. This plan consumes backend/src/orchestrator/gate.ts (runGate, GateResult, GateAttempt) and extends backend/src/test-units.ts, both created by Phase A. Build order: Phase A → BILTIQ-001.

Build Sequence

Each step = one logical commit. TDD where logic exists: RED → GREEN in the same step via npm run test:units (the repo's LLM-free harness). Commit format: BILTIQ-001: <step>.

Step 1 — Data model & typesM

Rationale: Everything downstream references these types; land them first.

Files: create backend/src/db/models/orchestrationRun.ts; modify backend/src/types.ts (add OrchestrationRun, PlanTask, OrchestrationEvent, TaskStatus, OrchestrationStatus).

Tests: none (type/schema decls — structural correctness exercised by Steps 2–3). Gate: cd backend && npm run typecheck — exit 0. (AC1 scaffolding)
Step 2 — Tool layer + result gateM

Rationale: The single tool wraps the existing pipeline; the result gate reuses Phase A. Pure-logic gate is the first testable unit.

Files: create backend/src/orchestrator2/tools.ts (ToolContext, ToolResult, Tool, runResearchPipelineTool wrapping ProspectOrchestrator, TOOLS registry), backend/src/orchestrator2/result-gate.ts (gradePipelineResult(run): GateResult).

Tests: gradePipelineResult — pass when run has a pitch and !lowConfidence; fail otherwise (AC6). Added to test-units.ts. Gate: npm run test:units — GREEN.
Step 3a — Engine skeleton + termination testsM

Rationale: The heart of the increment; DI makes the loop testable without an LLM (AC4). Land the loop + termination first.

Files: create backend/src/orchestrator2/engine.ts (OrchestrationEngine, EngineDeps = {planner, replanner, grader, synthesizer, tools}, the plan→execute→gate→replan→grade loop, maxIterations enforcement, event emission).

Tests (LLM-free, mock deps): stop-on-met (grader returns met → loop exits, goalMet=true); stop-on-cap (grader never met → exits at maxIterations, partial=true) (AC2, AC3). Gate: npm run test:units — GREEN.
Step 3b — Engine replan + status-transition testsM

Rationale: The adaptive behavior and task-state correctness, split out so each is an independently reviewable commit.

Files: modify backend/src/orchestrator2/engine.ts (apply replan output to the pending-task list; task status transitions).

Tests (LLM-free, mock deps): replan output replaces only pending tasks (done tasks untouched); status transitions pending→running→done and →failed on gate fail (AC4, AC6). Gate: npm run test:units — GREEN.
Step 4a — planner + replanner (LLM-backed)M

Rationale: Real plan/replan deps. Must route through llm/client.ts only (compliance boundary, Anti-Pattern #2).

Files: create backend/src/orchestrator2/planner.ts, backend/src/orchestrator2/replanner.ts using callWithTool; add ORCH_MAX_ITERATIONS to backend/src/config.ts.

Tests: extract + unit-test pure prompt-builders (buildPlannerPrompt, buildReplanPrompt) in test-units.ts; LLM calls covered by Step 8. Gate: npm run typecheck && npm run test:units.
Step 4b — goal-grader + synthesizer (LLM-backed)M

Rationale: The termination judge and final-answer assembler; separable commit from plan/replan.

Files: create backend/src/orchestrator2/goal-grader.ts (callWithTool), backend/src/orchestrator2/synthesizer.ts (callText).

Tests: unit-test pure prompt-builders (buildGraderPrompt, buildSynthesisPrompt) in test-units.ts; LLM calls covered by Step 8. Gate: npm run typecheck && npm run test:units.
Step 5 — Queue + workerM

Rationale: Run orchestrations off the request thread, mirroring the existing Run worker. Reuse runEventBus — do NOT build a second bus (Anti-Pattern #1).

Files: modify backend/src/queue/index.ts (add orchestration queue + enqueueOrchestration); modify backend/src/events/emitter.ts (add emitOrchestrationEvent/onOrchestrationEvent keyed orchestration:<id>, mirroring the run helpers); create backend/src/queue/orchestrationWorker.ts (build OrchestrationEngine with real deps, persist status/events like worker.ts); add worker:orchestration script to backend/package.json.

Tests: LLM-free unit test for onOrchestrationEvent/emitOrchestrationEvent round-trip (subscribe → emit → handler receives event, keyed by id) in test-units.ts; worker boot smoke-check (process starts and registers the queue without throwing). Gate: npm run typecheck && npm run test:units + worker boots.
Step 6 — Routes + SSEM

Rationale: Expose create/read/stream, mirroring routes/runs.ts conventions.

Files: create backend/src/routes/orchestrations.ts (POST /, GET /:id, GET / list, GET /:id/events SSE); register router in backend/src/server.ts at /api/orchestrations.

Tests: manual curl POST /api/orchestrations returns {orchestrationId}; SSE stream emits events (AC1). Gate: npm run typecheck.
Step 7a — Frontend types, API client, goal formM

Rationale: The data layer + entry point; lands before the live view that consumes it.

Files: modify frontend/src/lib/types.ts (orchestration types), frontend/src/lib/api.ts (client fns), frontend/src/components/Sidebar.tsx (Orchestrate entry); create frontend/src/app/orchestrate/page.tsx (goal form → POST → route to live view) (AC1).

Tests: cd frontend && npx tsc --noEmit — exit 0; manual: submit a goal, confirm redirect. Gate: frontend typecheck GREEN.
Step 7b — Live plan boardM

Rationale: The live view; reuses the run-detail SSE/poll pattern.

Files: create frontend/src/app/orchestrate/[id]/page.tsx + frontend/src/components/PlanBoard.tsx (task cards, child-Run links to /runs/[id], replan banner, partial badge — AC3, AC5).

Tests: cd frontend && npx tsc --noEmit — exit 0; manual: watch the board update live during a run. Gate: frontend typecheck GREEN.
Step 8 — ADRs + end-to-end integrationM

Rationale: Record the two architectural decisions; prove the whole loop on real data.

Files: create docs/adr/0002-orchestration-run-model.md, docs/adr/0003-orchestration-queue.md (drafted during planning — committed here).

Tests: live run — goal "Research Stripe and draft an outreach email" produces a child Run with a pitch and a non-empty finalAnswer; verify partial path with a deliberately hard goal + low maxIterations. Existing test-system.ts still passes (regression). Gate: manual acceptance against AC1–AC7.
