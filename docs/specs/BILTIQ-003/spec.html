Spec: BILTIQ-003 — Projects MVP (organize research by Project → Company → Run)

Task BILTIQ-003Owner @harishCreated 2026-06-23Wall-clock LCompliance on_prem_preferredParent Projects feature P1 of P1–P4

Problem

Research is organized by how it was started (a flat run list), not by who it is about. Users cannot see all of a company's runs together or group research into a campaign, so research is one-shot and disposable.

User Impact

Users create Projects, attach research to them, and browse a project's prospects grouped by company with each company's runs beneath. New research (both New Research and Orchestrate) can be filed under a project at creation, and "research again" on an existing run pre-fills a fresh run. The 14 existing runs remain, shown as "Unassigned".

Acceptance Criteria

  • OPENAC1 — A Project can be created (name + optional description), listed, and fetched; persisted in its own Mongo collection.
  • OPENAC2 — A run carries an optional projectId; POST /api/runs (New Research) and POST /api/orchestrations accept and persist it.
  • OPENAC3 — GET /api/projects/:id returns the project plus its runs grouped by lead.company (normalized), so the UI can render Company → Runs.
  • OPENAC4 — A Projects list page (create + list) and a Project detail page (companies, each expandable to its runs with links to /runs/[id]) exist; a "Projects" sidebar entry routes to them.
  • OPENAC5 — New Research and Orchestrate forms have an optional Project selector; selecting one files the run under it.
  • OPENAC6 — Each run card/detail exposes "Research again" that pre-fills a new run from that run's lead (surfacing the existing rerun path).
  • OPENAC7 — Runs with no projectId are reachable via an "Unassigned" grouping; no existing run is lost or mutated by the schema change.
  • OPENAC8 — Backend project model + the company-grouping helper are covered by LLM-free unit tests (grouping by normalized company; empty project).

Out of Scope (later increments / deferred)

  • Company-level chat over all runs (P2), build-on-prior pipeline seeding (P3), project-level synthesis/ranking (P4).
  • A first-class Company collection / cross-project company identity; company name de-dup/merge.
  • Team/multi-user, permissions, bulk move of runs between projects.

Open Questions

  • Resolved: 3-level hierarchy; Company is a derived grouping on normalized lead.company (no Company collection in v1); legacy runs → "Unassigned".
