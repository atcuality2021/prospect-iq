Design: BILTIQ-003 — Projects MVP

Approach

Add a Project Mongo collection and tag runs (and orchestrations) with an optional projectId. Company is a derived grouping — distinct normalized lead.company within a project — so no Company collection. A pure groupRunsByCompany helper powers the project-detail view. New pages: Projects (list+create), Project detail (companies → runs). New Research and Orchestrate forms gain a ProjectSelector. "Research again" surfaces the existing rerun path. Legacy untagged runs render under "Unassigned".

Alternatives Considered

  • Separate Company collection — rejected (v1): runs already carry lead.company; deriving avoids a sync problem.
  • Flat tags instead of hierarchy — rejected: user wants project→company→run structure.
  • Re-enqueue child runs per project — n/a: child runs inherit projectId via ToolContext.

Data / Compliance / Rollout

Additive: new projects collection + optional projectId on runs/orchestrations. No migration — the 14 existing runs stay, rendered "Unassigned" (AC7). Compliance unchanged (no new LLM/ external use; chat/seeding are later increments). Rollout: every new field/route is optional; existing flows behave identically when no project is selected.

Risks

  • Company-name ambiguity ("Stripe" vs "Stripe Inc") → group on normalizeCompany (trim+lowercase); accept imperfect grouping in v1.
  • Threading projectId into child runs touches engine/tools — keep it a single optional field on ToolContext; default undefined preserves Inc-1/2 behavior.
